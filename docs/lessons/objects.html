1. Introduction to objects

Much earlier in the course, when we introduced data types, we mentioned that there existed primitives, which could be expressed with a number, and objects, which were much more complex. We added that strings were our first type of object: being a sequence of character, no one number could adequately represent a string. Instead, we would store the address (a numerical value) of the location of the sequence of characters in memory.

Later, when we introduced collections, we mentioned that they were too complex to express as single numbers as well, and that just like strings, we would store the base address of our collections in memory and use indices to navigate from that base address to each value stored.

Now that we have a broader understanding of Python, we can complement our definition of objects:

Objects are complex structures that can contain variables and that can possess functions that define what it can do and how it can interact with internal and external data. The variables posessed by objects are called instance variables or attributes, and the functions are called methods.

2. Looking back at strings

For our purposes, we will not talk much of the instance variables belonging to strings. However, they do have a number of useful methods that can help you manipulate strings depending on your needs:

Method Effect
lower() Returns the string with all its characters converted to lowercase.
upper() Returns the string with all its characters converted to uppercase.
split() Splits a string into a list of characters and returns it.
strip() Removes space at the beginning and ending on the string and returns it.
replace(x,y) Replaces all the instances of x with y in the string and returns it.
find(s) Returns the index of the first occurrence of s in the string

To use any of these methods, you should always be mindful of applying it on a string: to call split() on a string variable S, for example, you would use S.split(). This differs from how we used functions in the past, which were used alone.

3. Creating objects

Now that we have a formal definition of what objects are, we need to look at how they are created. Objects are meant to be customizable structures through which you can implement new functionality adapted to the problems you are tackling. As such, we will combine what we have learned so far to produce them.

3.1 Classes

Classes are the containers that will hold our objects’ code. Much like other blocks, they possess scope and have a keyword associated with them: class:

class MyObject():
  #My object's code

We will not get into what classes can do for us beyond containing object definitions, but if you delve deeper into Python, you’ll see that they can do a lot more than that.

Note that just like other blocks, the code contained within a class is indented: failure to indent will result in errors. Similarly, scope applies to the class blocks, and anything declared inside of a class is only valid within that class.

The code we will store inside of classes is called a definition. By itself, an object’s definition simply outlines the blueprint of the object. We will see later that we can use that blueprint to produce our usable objects.

3.2 Constructors

The bare minimum required to build an object is a special method called a constructor. We have seen constructors before, when we talked about functions like str(), list() and dict(): their role is to initialize objects so that they are ready for use. Based on what we said about blueprints, we can now say that constructors will “follow the plan” outlined in the object definition and create an object instance.

Object instances are nothing more than “one copy of a structure that follows an object’s definition”. For example, "Hello" and "World" are both string instances: they are both strings as they follow the same blueprint, despite having different content.

To define our constructor, we will proceed the same way we did to create other functions: by using def:

class MyObject():
  def __init__(self):
    #My constructor's code

Our constructors always have the same name: __init__: this identifies them as a “special entity”. This name also makes it easier to remember what they do because they initialize an object – any code in the constructor is executed when the object is created and only then. This makes it a perfect place to create instance variables and prepare any structure that needs setting up.

In most cases, constructors will be used to create instance variables, which can be done through the self keyword included as the first parameter of our constructor. self refers to the “current object”, and can be used to execute other methods (as we’ll see in the next section) or to attach instance variables to our object.

For example, we can create two instance variables in our MyObject object by adding the following to our constructor:

class MyObject():
  def __init__(self):
    self.colour = "Blue"
    self.size = 2

By doing this, we are creating two instance variables within each object of type MyObject and we give them static values.

Once this is done, we can create an object of type MyObject at any time by typing:

my_obj = MyObject()

Note the similarity with the way we created lists through list().

Of course, constructors are methods, which themselves are functions: input parameters can be added to their signature:

class MyObject():
  def __init__(self, colour, size):
    self.colour = colour
    self.size = size

With this definition, we can now define the values of each instance’s colour and size instance variables. The new constructor can be called using MyObject("some colour", 5) or any variation thereof.

It’s very important to note that the self input parameter doesn’t need to be given to our constructor: it is implicitly defined by the object.

3.3 Instance variables

As we saw earlier, instance variables are variables that belong to the object. As such, they can be described as attributes or properties. To access an instance variable, you can use the self keyword introduced earlier – this can be done in the same manner as with any other variable.

For cleanliness, it’s preferable to only create instance variables in the constructor, but they can be accessed and modified in any method (and even, less preferably, from outside the object):

#Using the previous definition of `MyObject`

obj = MyObject("Blue", 5)

#Accessing them from the outside
print(obj.size)
print(obj.colour)

obj.colour = "Red"

print(obj.colour)

To use them from methods, we need to add non-constructor methods to our object definition, which we will do below.

3.3 Methods

Methods are functions that can be executed on our objects. They are defined within the object’s definition and allow us to implement and package functionalities within our object to combine data storage and manipulation.

Let’s revisit our previous definition of MyObject:

class MyObject():
  def __init__(self, colour, size):
    self.colour = colour
    self.size = size

To add new methods to our object definition, we can append new def blocks that mimic the constructor:

class MyObject():
  def __init__(self, colour, size):
    self.colour = colour
    self.size = size

  def print_everything(self):
    print(self.colour, self.size)

To ensure that our methods have access to our instance variables, all methods have the implicit parameter self as first input parameter. This allows the inside of the method to feature accesses and mutation using the self keyword.

In the example above, our print_everything method takes no arguments per se, only the implicit self, and uses both of our instance variables to print output to the screen.

Much like our list methods, we can use MyObject’s methods by instantiating a variable of type MyObject then calling the method on it:

#Using the previous definition of `MyObject`

obj = MyObject("small", 5)
obj.print_everything()
#Output: "'small' 5"

The same principles apply to methods as to normal functions. As such, you can append as many input parameters as needed to your methods to inject information in your object or to mutate the object depending on outside information. As such, we could also have a method that updates the colour and size instance variables:

class MyObject():
  def __init__(self, colour, size):
    self.colour = colour
    self.size = size

  def print_everything(self):
    print(self.colour, self.size)

  def update_everything(self, colour, size):
    self.colour = colour
    self.size = size

Note that this is an example of why using self is important: in the update_everything method, we have two variables named colour: one is local (i.e. created in the method, but not an instance variable), and the other is an instance variable attached to self. The former has a scope limited to the method, the second exists as long as the object it is contained in does.

4. Modeling example: The library

The main purpose of objects is to allow programmers to define data structures that go beyond simple lists and dictionaries and that address specific needs. In this example, we’ll be designing a small program that can handle library inventory and bookings using objects and built-in data structures.

4.1 Requirements

Before we start coding, we need to list out the requirements of our program to know what to aim for and to avoid overcomplicating our design. Our requirements should outline the structures that we need and the actions that the program needs to be able to perform.

Structures

  • A book: a book holds information such as its title, its author, its availability and even the last time it was checked out. We’ll concentrate on the first two parameters and leave the last as an exercise.
  • A library: a library contains many books and handles actions that users can execute on the books.

Note, we could design our library program further to include shelves and whatnot, the level of complexity really depends on the functionalities you want to implement

Actions

  • A user must be able to look for books in the library’s collection
  • A user must be able to borrow a book that is available
  • A user must be able to put back a book that has been borrowed

The structural requirements (under structure) give us guidance on which _objects__ we will implement, and the actions give us information on the methods that we might need.

4.2 Modeling books

As stated earlier, books need at least three properties: a title, an author and whether they are available at the moment or not. Since the book cannot do anything by itself, it will not need any methods past its constructor. As such, we can build from our previous MyObject example and write the following definition:

class Book():
  def __init__(self, title, author):
    self.title = title
    self.author = author
    self.available = True

The Book class is mainly meant to store data. We could of course add more information and even methods, but that would go beyond our design. Note that we do not need to customize the available instance variable: when created, the Book is available and ready to be borrowed.

4.3 Modeling the library

The library has a lot more moving parts. It will not only contain many books, but also allow actions to be performed on the books it has.

Based on this, we conclude that the Library object needs at least one instance variable: a list containing our books! Note that we could also use a dictionary; try thinking of what advantage using a dictionary would give us over using a simple list.

class Library():
  def __init__(self):
    self.books = list()

From there, we can look at adding methods to be able to create books. For cleanliness, the user should only interact with the Library object (as they would in real life), as such, we can add a add_book method to our Library:

#Cont'd from previous snipper

  def add_book(self, title, author):
    book = Book(title, author)
    self.books.append(book)

Note that we use our constructor implemented in Book to cleanly create a new instance of the Book object before storing the resulting object in our list, which symbolizes “the shelves.”

From this point on, we can create a Library and add as many books to it by using add_book couple with a title and an author name.

Seeing the current catalog would be useful as well: since our catalog is a list, we can iterate through it, printing each book’s data:

#Cont'd

  def see_all(self):
    for book in self.books:
      print("Title: " + book.title + " Written by: " + book.author)
      
      if book.available:
        print("CURRENTLY AVAILABLE!")
      else:
        print("UNAVAILABLE")

Using this, we can call see_all() and get an overview of our library!

Usage would follow the model we established for MyObject:

my_library = Library()

my_library.add_book("Oliver Twist", "Dickens")
my_library.add_book("Harry Potter", "J.K. Rowling")

#Prints all the book titles/authors
my_library.see_all()
